# Архитектура проекта

## Основная идея

Проект организован по схеме `core/ + tabs/`.

- `tabs/<имя_вкладки>/` — **feature package** одной вкладки (UI + бизнес-логика этой вкладки).
- `core/` — только общий код: инфраструктура, общие утилиты, общий домен.
- `streamlit_app.py` — тонкий composition root: конфигурация, bootstrap, маршрутизация вкладок.

## Правила размещения кода

1. Если код используется только одной вкладкой — он должен жить в `tabs/<name>/`.
2. В `core/` переносится только код, который:
   - используется минимум в двух вкладках; или
   - является инфраструктурой приложения.
3. Нельзя создавать новый «общий свалочный» модуль вида `core/utils.py`.
4. Плоские legacy-модули верхнего уровня и transitional `utils/*` удалены намеренно.

## Границы и ответственность

- `core/app/` — bootstrap, шапка, admin-short-circuit, контекст приложения.
- `core/db/` — доступ к данным и константы источников данных.
- `core/lineage/` — граф родословной и нормализация имён.
- `core/ui/` — повторно используемые UI-хелперы и рендереры.
- `tabs/*` — независимые feature-пакеты вкладок.

## Эволюция архитектуры

- Любой перенос кода из вкладки в `core/` должен обосновываться повторным использованием.
- Изменения хранилища (например, CSV → SQLite) должны изолироваться за `core/db`.
- Публичные query params и share-link payload не должны меняться без отдельной миграции.

## Языковое правило

Для поддержания единообразия docstrings и комментарии в затрагиваемых файлах пишутся на русском языке.
